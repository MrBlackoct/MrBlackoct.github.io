AJAX

发表于 2018-02-09

AJAX

什么是AJAX

  1. Ajax是Asynchronous Javascript and XML的缩写.是指一种创建交互式网页应用的网页开发技术.
  2. Ajax包含一下技术
    • 基于web标准的XHTML + CSS的表示
    • 使用DOM进行动态显示及交互
    • 使用XML和XSLT进行数据交换及相关操作
    • 使用XMLHttpRequest进行异步数据查询/检索
    • 使用Javascript将所有东西都绑定在一起
  3. Ajax的优势
    • 通过异步模式,提升了用户体验
    • 优化了浏览器和服务器之间的传输,减少不必要的数据往返,减少了带宽占用
    • Ajax引擎在客户端运行,承担了一部分本来由服务器承担的工作,从而减少了大用户量下的服务器负载
  4. Ajax最大的特点
    Ajax可以实现动态不刷新(局部刷新).就是能在不更新整个页面的前提下维护数据.这使得web应用程序更为迅捷地回应用户动作,并避免了在网络上发送那些没有改变过的信息.

Ajax技术体系的组成部分

HTML / CSS / DOM / XML / XMLHttpRequest / Javascript

Ajax的CALLBACK

共有八种 :
onSuccess / onFailue / onUninitialized / onLoading / onLoaded / onInteractive / onComplete / onException

Ajax的优缺点

  1. 优点
    • 最大的一点是页面无刷新,用户体验好
    • 使用异步方式与服务器通信,具有更加迅速的响应能力
    • 可以把以前一些服务器负担的公告你做转嫁到客户端,利用客户端限制的 能力来处理,减轻服务器和带宽的负担,节约空间和带宽租用成本.并且减轻服务器的负担,ajax的原则是”按需取数据”,可以最大程度地减少冗余请求和响应对服务器造成的负担.
    • 基于标准化的并被广泛支持的技术,不需要下载插件或小程序.
  2. 缺点
    • 不支持浏览器的回退按钮
    • 安全问题,Ajax暴露了与服务器交互的细节
    • 对搜索引擎的支持比较弱
    • 破坏了程序的异常机制
    • 不容易调试

XMLHttpRequest对象的常用属性和方法

  • open(‘method’,’URL’)
    建立对服务器的调用,第一个参数是HTTP请求,方式可以为GET或POST.第二个参数是请求页面的URL.
  • send()
    发送具体请求
  • abort()
    停止当前请求
  • readyState属性
    请求的状态,有5个可取值.
    0 = 请求未初始化
    1 = 服务器连接已建立
    2 = 请求已接受
    3 = 请求处理中
    4 = 请求已完成,且响应已就绪
  • status
    服务器的Http状态码
    200 = OK / 404 = 未找到 Not found
  • responseText属性
    服务器的响应,表示为一个串
  • responseXML属性
    服务器的响应,表示为一个XML

TCP长连接&短连接&HTTP长轮询&段轮询

TCP长连接 & 短连接 / &HTTP长轮询 & 段轮询

TCP长连接 & 短连接

  1. 短连接
    每次HTTP请求都会建立TCP连接,管理容易.
  2. 长连接
    只需要建立一次TCP连接,以后HTTP请求重复使用同一个TCP连接,管理难.

长轮询 & 段轮询

  1. 短轮询
    客户端定时像服务器发送Ajax请求,服务器接到请求后马上返回响应信息并关闭连接.
  • 优点 :
    后端程序编写比较容易
  • 缺点 :
    请求中打扮时无用&浪费带宽和服务器资源.
  • 实例 :
    适用于小型应用
  1. 长轮询
    客户端像服务器发送Ajax请求,服务器接到请求后hold住连接,直到由新消息才返回响应信息关闭连接,客户端处理完响应信息后再向服务器发送新的请求.
  • 优点 :
    在无消息的情况下不会频繁地请求,耗费资源小
    缺点 :
    服务器hold住连接会小号资源.返回数据顺序无保证,难于管理维护.

三次握手 & 四次挥手

发表于 2018-02-09

三次握手 & 四次挥手

三次握手(由客户端执行connect来触发)

  1. Client将标志位SYN置为1,随机长生一个值seq = J,将该数据包发送给Server,Client进入SYN_SENT状态,等待Server确认.
  2. Server收到数据包后由SYN = 1知道Client请求建立连接,Server将标志位SYN和ACK都置为1,ack(number) = J + 1,随机产生一个值seq = k,将数据包发送给Client以确认连接请求,Server进入SYN_RECV状态.
  3. Client收到确认后,检查ack是否为J + 1,ACK是否为1.如果正确,则将标志位ACK置为1,ack = k + 1.将数据包发送给Server,Server检查ack是否位k + 1,如果正确则连接建立成功,Client和Server进入ESTABLISH状态,完成三次握手,随后Client和Server之间可以传输数据.
  4. 三次握手.jpg

四次挥手(由客户端 / 服务端执行close触发)

由于TCP连接时全双工的,一次每个方向都必须单独进行关闭.这一原则是当一方完成数据发送任务后,发送一个FIN来终止这一方向上的连接,收到一个FIN只是意味着这一方向上没有数据流动了,即不会再收到数据了,但是这个TCP连接上仍然能够发送数据,直到这一方也发送了FIN.首先进行关闭的一方执行主动关闭,另一方执行被动关闭.

  1. Client发送一个FIN,用来关闭Client到Server的数据传输,Client进入FIN_WAIT_1状态.
  2. Server收到FIN后,发送一个ACK给Client,确认序号为收到序号+1.Server进入CLOSE_WAIT状态.
  3. Server发送一个FIN,用来关闭Server到Client的数据传输,Server进入LAST_ACK状态.
  4. Client收到FIN后,进入TIME_WAIT状态,接着发送一个ACK给Server,确认序号为收到序号+1,Server进入CLOSED状态.
  5. 四次挥手.jpg

为什么建立连接时三次,关闭是四次?

这是因为服务端在LISTEN状态下,收到建立连接的请求报文后,把ACK和SYN在一个报文里发给客户端.而关闭连接时,当收到对方的FIN报文时,仅仅标识对方不在发送数据了,但是还能接收数据,己方数据也未必全部发给对方了,所以己方可以立即close,也可以发送一些数据之后,在发送FIN报文给对方,标识统一现在关闭连接.因此,己方ACK和FIN一般都会分开发送.

Promise & 伪数组

Promise

  • Promise对象用于一个异步操作的最终完成(或失败)及其结果值的表示.本质上,一个Promise是某个函数返回的对象,可以把回调函数绑定在这个对象上,而不是把回调函数当作参数传进函数.
  • 保证 :
  1. 在JS事件队列的当前运行完成之前,回调函数永远不会被调用.
  2. 通过 .then()形式添加的回调函数,甚至都在异步操作完成之后才被添加的函数都会被调用.
  3. 通过多次调用.then(),可以添加多个回调函数,它们会独立地按照插入顺序执行.
  • 语法
    new Promise(function(resolve, reject)){...}
    executor是一个带有resolve / reject两个参数的函数.executor函数在Promise构造函数执行时同步执行,被传递resolve和reject函数(executor函数在Promise构造函数返回新对象之前被调用).resolve和reject函数被调用时,分别将promise的状态改为fullfield(完成) / reject(失败).executor内部通常会执行一些异步操作,一旦完成,可以调用resolve函数来将promise炸u那个太改成fullfield或在发生错误时调用reject改为rejected.如果在executor函数中抛出一个错误,那么该promise状态为rejected.executor函数的返回值被忽略.
  • 链式调用

Javascript的垃圾回收机制与内存管理 & Javascript值的传递方式

发表于 2018-02-06

Javascript的垃圾回收机制与内存管理 & Javascript值的传递方式

Javascript的垃圾回收机制与内存管理

  1. 原理
    垃圾回收器会定时(周期性)找出那些不在继续使用的变量,然后释放其内存.
  2. 内存泄露
    是指程序中已动态分配的堆内存由于某种原因程序未释放或无法释放,造成系统内存的浪费,导致程序运行速度减慢甚至系统崩溃等严重的后果.
  3. Javascript的垃圾回收机制
    找出不再使用的变量–局部变量,闭包除外.然后释放掉其内存,但是这个过程不是实时的,因为其开销比较大,所以垃圾回收器会按照固定时间间隔周期性地执行.
    全局变量的生命周期直到浏览器卸载页面才会结束.
  • 标记清除
    垃圾回收器会在运行时给存储在内存中的所有变量加上标记(方式不限).然后,它会去掉环境中的变量以及被环境中变量引用的变量(闭包).再次之后再被加上标记的变量将被视为准备删掉的变量,原因是环境中的变量已经无法访问到这些变量了.最后,垃圾回收器完成内存清除工作,销毁那些代表及的值并回收它们所占用的内存.
  • 引用计数
    引用计数的含义是跟踪记录每个值引用的次数.当声明了一个变量并将一个引用类型复制给该变量时,则这个值的引用次数就是1.如果同一个值又被付给另一个变量,则引用次数 +1 .相反,如果包含对这个值引用的变量又取得了另外一个值,则引用次数 -1 .当这个值的引用次数变为0时,则说明没有办法再访问这个值了.下次运行垃圾回收器时将它回收.
    • 循环引用 :
      对象A中包含一个指向B的指针,B中包含一个指向A的指针,会造成计数不为0而引起的内存泄漏
  • Javascript引擎基础GC方案(标记清除)
    1. 遍历所有可访问对象
    2. 回收已不可访问对象

Javascript值的传递方式

  • 基本类型都是按值传递的

  • 对象的值按共享传递
    调用函数传参时,函数接受对象实参引用的副本.它和按引用传递的不同在于 : 在共享传递中对函数形参的对象是相同的.共享相同的对象,所以修改形参对象的属性值,也会影响到实参的属性值.

  • 个人理解 :
    就好比小王 / 小李共同牵着同一条狗,小王剪头发不会影响到小李的发型,但是小李给狗剃毛会造成小王牵的狗也秃了.

HTTP缓存机制与原理

  1. 为方便理解,认为浏览器存在一个缓存数据库,用于储存信息.在客户端第一次请求数据时,此时缓存数据库中没有对应的缓存数据,需要请求服务器,服务器返回后,将数据储存至缓存数据库中
  2. 分为两大类
    强制缓存 / 对比缓存
    1. 已存在缓存数据时 & 强制缓存
    • 缓存命中 :
      客户端 缓存数据库
      | |
      | 请求数据 |
      |—————————————————————–>|
      | |
      | 有缓存数据,且未失效,返回数据 |
      |—————————————————————–>|
      | |
      客户端 缓存数据库
    • 缓存未命中 :
      客户端 缓存数据库 服务器
      | | |
      | 请求数据 | |
      |———————————>| |
      | 缓存数据失效 | |
      |<———————————| |
      | 请求|数据 |
      |———————————-|——————————>|
      | 返回数据|和缓存规则 |
      |<———————————|——————————-|
      | 将数据和缓存规则存入缓存系统 | |
      |———————————>| |
      客户端 缓存数据库 服务器
    1. 已存在缓存数据时 & 对比缓存
    • 缓存命中 :
      客户端 缓存数据库 服务器
      | | |
      | 获取缓存数据标识 | |
      |———————————>| |
      | 返回缓存数据标识 | |
      |<———————————| |
      | 请求服务器验证缓存标|识对应数据是否失效 |
      |———————————-|——————————>|
      | 通知客户端|缓存未失效 |
      |<———————————|——————————-|
      | 获取缓存数据 | |
      |———————————>| |
      | 返回缓存数据 | |
      |<———————————| |
      客户端 缓存数据库 服务器
    • 缓存未命中 :
      客户端 缓存数据库 服务器
      | | |
      | 获取缓存数据标识 | |
      |———————————>| |
      | 返回缓存数据标识 | |
      |<———————————| |
      | 请求服务器验证缓存标|识对应数据是否失效 |
      |———————————-|——————————>|
      | 返回最新数据|和缓存规则 |
      |<———————————|——————————-|
      | 将数据和缓存规则存入缓存系统 | |
      |———————————>| |
      客户端 缓存数据库 服务器
  3. 对于强制缓存来说,响应header中会有两个字段来表明失效规则(Expires / Cache-Control)
    • Expires
      现在基本淘汰,是服务器返回的到期时间.
    • Cache-Control
      pravite : 客户端可以缓存
      public : 客户端和代理服务器可以缓存
      max-age = xxx : 缓存内容将在xxx秒后失效
      no-cache : 需要使用对比缓存验证数据
      no-store : 所有内容都不缓存
  4. 对比缓存
    • 浏览器第一次请求数据时,服务器会将缓存数据与标识一起返回给客户端,客户端将二者备份只缓存数据库中.
    • 再次请求数据时,客户端将备份的缓存标识发送给服务器,服务器根据缓存标识进行判断,判断乘公共后,返回304状态码.通知客户端比较成功,可以使用缓存数据.
    • 对比生效时,只返回header部分,通过状态码通知客户端使用缓存,大大节省时间.
    • 两种标识
    1. Last-Modified / If-Modified-Since
      • Last-Modified :
        服务器在相应请求时,告诉浏览器资源的最后修改时间.
      • If-Modified-Since
        再次请求服务器时,通过此字段通知服务器上次请求时,服务器返回的资源最后修改时间.
      1. 若资源的最后修改时间大于If-Modified-Since,则说明资源又被改动过,则响应整片资源内容,返回状态码200.
      2. 若资源的最后修改时间小于或等于If-Modified-Since,说明资源无修改,响应304,告知浏览器继续使用所保存的cache.
    2. Etag / If-None-Match(优先级高)
      • Etag :
        服务器响应请求时,告诉浏览器当前资源在服务器的唯一标识(生成规则由服务器决定)
      • If-None-Match
        再次请求服务器时,通过此字段通知客户端缓存数据的唯一标识.
      1. 不同,说明资源又被改动过,则响应整片资源内容,返回200.
      2. 相同,说明未再改动,响应304,告知浏览器继续使用所保存的cache.

事件模型及事件代理

事件模型及事件代理

事件模型

  1. 原始事件模型(DOM0级)
    一种被所有浏览器支持的事件模型.对于原始事件而言,没有事件流,事件一旦发生马上进行处理.
    缺点 :
    • 逻辑与显示没有分离
    • 相同事件的监听函数只能绑定一个,后面绑定的会被前面覆盖掉.
    • 无法通过事件的冒泡 / 委托等机制完成更多的事情.
  2. DOM2级
    三个过程(一次事件的发生)
    1. 事件捕获阶段 :
      当某个元素触发某个事件,顶层对象document就会发出一个事件流,随着DOM节点向目标元素节点流去,知道到达事件真正发生的目标元素.此过程中,事件相应的监听函数不会被触发.
    2. 事件目标 :
      当到达目标元素之后,执行目标元素该事件相应的处理函数.如果没有监听函数则不执行.
    3. 事件冒泡 :
      从目标元素开始,往顶层传播.途中如果节点绑定了相应的事件处理函数,这些函数都会被一次触发
    • 除了blur / focus / load / unload之外,其他事件都可以冒泡.
  3. IE事件模型
    IE不把该对象传入事件处理函数,由于任意时刻只会存在一个事件,所以IE把它作为全局对象那个Windows的一个属性.
    IE的事件模型只有两步 :
    先执行元素的监听函数,然后事件沿着父节点一直冒泡到document.

事件代理

  1. 原理
    将事件处理器绑定到一个父级元素上,避免了频繁绑定多个子级元素,依靠事件冒泡与事件捕获机制,子级元素的事件委托给父级元素.
  2. 优点
    • 可以大量节省内存占用,减少事件注册
    • 可以实现再新增子对象时无需再次对其金星绑定事件,对动态内容尤为合适
  3. 缺点
    如果把所有事件都代理,就可能出现误判.

监听事件的绑定与移除

  1. 绑定
    addEventListener
    element.addEventListener(type, handle, false / true)
    type : 事件类型
    handle : 事件执行触发的函数
    false / true : 冒泡 / 捕获. 表示在哪个阶段调用事件处理程序
  2. 移除
    removeEventListener
    element.removeEventListener
    参数值同上

如何阻止事件冒泡

W3C中 : stopPropagation()
IE中 : cancleBubble = true

解决项目中莫名其妙跳出多次弹窗提醒问题的过程

发表于 2018-01-09

解决项目中莫名其妙跳出多次弹窗提醒问题的过程

  1. 背景简介
    Project模块下有多个子模块,分别为newProject / penddingProject / myProject / projectInfo 等,其中整体模块由projectController.js控制,其余子模块有各自对应的Controller.各子模块由switchtab控制显示,不切换路由,各tab可同时存在于页面上.
    在newProject模块中,有表单保存功能,表单引用app-iframe组件.在加入保存成功后弹窗提醒功能后,测试中发现会弹出多次”保存成功”提醒,或两次,或四五次,甚至六七次.
    //使用AngularJS作为开发框架

  2. 逐步排查

    1. 首先检查alert的逻辑与数量.利用在newprojectcontroller.js中使用$scope.$on()监听接受广播消息SAVE_FORM_SUCCESS确认保存成功后弹窗提醒.检查后确认逻辑与数量都无误.

    2. 继续尝试弹出次数虽多但不定的情况.发现如果页面中存在多个子模块页面即存在多个app-iframe组件时提醒次数则会增多.初步判断可能是因为多个app-iframe之间会相互影响.检查app-iframe.js后发现广播SAVE_FORM_SUCCESS的机制只是判断了接收到的消息的类型,然后选择广播VALIDATE_FORM_SUCCESS / SAVE_FORM_SUCCESS.所以当页面中加载了多次app-iframe组件时就会广播多次该消息.

    3. 修改代码,为广播SAVE_FORM_SUCCESS部分加入判断.$rootScope.isListenedSavaSuccess为自定义变量,第一次判断时其一定为false,进入判断后将其置为true,则以后不会再进入改循环,然后正常进行广播事件的判断与执行.这样解决了调用一次app-iframe.js时会因为加载了多个app-iframe而出现多次广播的问题.

    4. 打断点继续测试,当加载多个app-iframe时确实只广播了一次SAVE_FORM_SUCCESS.但是发现仍然弹窗提醒两次,打断点发现alert部分被调用了两次.原因是newProject模块和myProject模块中的申请新流程页面都是加载newProjectController来实现的,所以$scope.$on()两次调用.

    5. 将$scope.$on()部分提升到projectController.js处,由于这是整个project模块的父级控制器,所以无论何时只会做出一次监听动作.至此问题解决.

  3. 总结
    遇到莫名其妙的问题后应根据其特征与规律入手,逐项检查,利用好断点调试可以监测程序的执行过程中你想要得到细节的部位的细节.

继承

约定父类:

1. 原型继承(将父类实例作为子类原型)

  • 实现

  • 特点 :

    1. 实例是父类和子类的共用实例
    2. 父类新增原型/实例方法,子类能都访问到
  • 缺点 :
    1. 要想为子类新增方法,必需要在new A()这样的语句前执行,不能放在构造器中
    2. 无法实现多继承
    3. 来自原型对象的引用属性是所有实例共享的
    4. 创建子类实例时,无法向父类构造函数传参

2. 构造继承(使用覅类构造函数来增强子类实例,等于赋值父类属性给子类)

  • 实现 :

  • 特点 :

    1. 解决了子类实例共享父类属性的问题
    2. 创建子类实例时可以向父类床底参数
    3. 可以实现多继承
  • 缺点 :
    1. 实例并不是父类的实例,只是子类的实例
    2. 只能击继承父类的实例属性和方法
    3. 无法实现函数复用,每个子类都有父类实例函数的副本,影响性能

3. 实例继承(为父类实例增加新特性,作为子类实例返回)

  • 实现 :

  • 特点 :

    1. 不限制调用方式,不管是new子类()还是子类(),返回的对象都具有相同效果
  • 缺点 :
    1. 实例是父类的实例,不是子类的实例
    2. 不支持多继承

4. 拷贝继承

  • 实现 :

  • 特点 :

    1. 支持多继承
  • 缺点 :
    1. 效率低,占用内存高
    2. 无法获取父类不可枚举的方法

5. 组合继承(通过父类构造,继承父类的属性并保留传参的特点.然后通过父类实例作为子类原型,实现函数复用)

  • 实现

  • 特点 :

    1. 弥补了方式2的缺陷,可以继承实例属性/方法,也可以继承原型属性/方法.
    2. 既是子类的实例,也是父类的实例
    3. 不存在引用属性共享问题
    4. 可传参
    5. 函数可复用
  • 缺点 :
    1. 调用了两次父类函数,生成了两份实例(子类实例覆盖了子类原型上的那份)

6. 寄生组合继承(通过寄生方式,砍掉父类属性)

  • 实现

  • 特点 :

    1. 堪称完美
  • 缺点 :
    1. 实现复杂
